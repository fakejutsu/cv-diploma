# SDD (Specification-Driven Development)

Этот каталог содержит артефакты SDD — процесса разработки через формализацию решений и синхронизацию спецификации с кодом.

## Текущая структура

- `CURRENT_SPEC.md` — актуальное состояние системы (без истории).
- `<iteration_number>_<short_name>/iteration.md` — завершённая итерация.

Текущая последняя итерация:
- `003_swin_stride_contract_fix/iteration.md`

## Правила итераций

- Номер итерации: zero-padded (`001`, `002`, ...).
- Имя: lowercase + underscore.
- Одна итерация = одна логическая задача.
- Старые итерации не переписываются.

Каждая итерация должна содержать:
- `Goal`
- `Inputs`
- `Findings`
- `Decisions`
- `Spec updates`
- `Open questions`
- `Next iteration`

## Жизненный цикл

1. Создать новую итерацию (следующий номер).
2. Зафиксировать цель.
3. Проверить код/контракты.
4. Принять решения и обновить спецификацию.
5. Сверить, что код и SDD не расходятся.

## Политика синхронизации

Если код и SDD расходятся, SDD считается устаревшим и должен быть обновлён.

## Границы применения

SDD обязателен для:
- изменений архитектуры,
- изменений training pipeline,
- изменений контрактов моделей,
- изменений форматов данных.

SDD можно опустить для:
- мелких багфиксов без изменения поведения,
- рефакторинга без изменения контрактов,
- изменений только комментариев/логов.
